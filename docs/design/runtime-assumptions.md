# 런타임 가정

이 스크리너는 **24/7 켜져 있지 않을 수 있는 호스트**(예: 개인 PC/노트북)에서 돈다고 가정한다.
sleep/wake/종료가 잦을 수 있다는 전제가 스케줄·복구 설계의 근거다.

## 설계에 미치는 영향

- **고정 시각 cron 가정 금지** — "매주 토요일 08:00" 같은 단일 시각 잡은, 그 시간에 호스트가
  켜져 있다는 보장이 없어 폐기했다. 대신 **hourly 잡 + 부팅 시 복구** 조합을 쓴다
  ([큐와 자동 복구](queue-and-recovery.md)).
- **누락된 시각은 정상** — 호스트가 꺼져 있던 동안의 정각 잡은 그냥 건너뛴 것으로 본다.
  부팅 시 `cleanup_stale_runs` + `reset_stuck_queue` + `reset_failed_to_pending`가 복구한다.
- **장시간 hang 진단** — 분석이 오래 멈춘 것처럼 보이면, 코드/SDK 버그를 의심하기 전에
  **호스트가 그 사이 sleep 상태였을 가능성**을 먼저 고려한다. `ended_at`이 부팅 시점의
  `crashed` 마킹일 수 있어, 실제 await 시간과 프로세스 정지 시간을 로그만으로 구분하기 어렵다.
- **timeout 방어 유지** — sleep 복귀 후 SDK 소켓이 끊긴 채 무한 await하는 케이스를 끊기 위한
  서브에이전트 timeout 방어 코드는 그대로 유지한다.

## 관련 주의

- `pythonw` + VBS redirect 조합이 sleep/wake 후 로그를 0바이트로 남기는 현상이 관찰된 적 있음
  — 로그 기반 진단이 안 될 때 이 조합을 의심.
