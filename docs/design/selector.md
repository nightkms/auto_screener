# 종목 선정 정책 (selector)

`selector.py`의 hot pick 선정은 **3단계**로 동작한다.

1. **1차 — 검색 상위**: 네이버 검색 상위(`finance.naver.com/sise/lastsearch2.naver`) 약 30개 중
   dedup 통과분을 **검색 순위 그대로** `top_n`개 선택.
2. **1.5차 — 상한가·거래량 급증**: 1차가 모자라면 부족분을 시세 리스트 페이지
   (`sise_upper.naver` 상한가 → `sise_quant.naver` 거래량)에서 dedup 통과분으로 채운다.
   우선순위는 **상한가 > 거래량**.
3. **2차 — z-score 보강**: 그래도 모자라면 부족분만 시총 상위 200 풀에서 z-score 룰로 채움.
4. **0건 허용**: 모든 단계가 dedup에 걸리면 **강제로 채우지 않고 0건을 반환**한다.

## 설계 의도

- 목표는 "시총·추세 상위"가 아니라 **사람들이 오늘 실제로 검색·관심 갖거나 실제로 크게
  움직인 종목**이다. 검색 상위는 등락률·테마·뉴스 카탈리스트가 압축된 신호, 상한가·거래량
  급증은 "오늘 실제로 움직인" 보조 신호로 본다.
- **dedup 고갈 대응**: 검색 상위·시총 풀은 거의 고정이라 자주 돌리면 30일 dedup에 다 걸려
  후보가 마른다. 상한가·거래량 급증은 **매일 명단이 크게 바뀌므로** dedup을 통과할 새 종목을
  꾸준히 공급한다(1.5차 도입 배경).
- dedup은 의도적으로 유지한다(다양한 종목 커버리지가 목적). 어떤 소스든 최근 30일 내
  분석된 종목이면 skip. 따라서 **0건은 버그가 아니라 정책의 자연스러운 결과**다.

## 유지보수 주의

- 1차(검색) → 1.5차(상한가·거래량) → 2차(z-score) 보강의 순서를 깨지 말 것.
- dedup 윈도우(30일, `recently_analyzed_tickers(days=30)`)는 자동으로 줄이지 말 것.
- 0건 결과를 오류로 처리하지 말 것 — 로그 한 줄로 끝내고 다음 정각에 재시도한다.
- 검색 상위 페이지는 ~28종목만 노출된다(`limit 30`이어도 그 이상은 못 가져옴).
  더 필요하면 다른 페이지/API 발굴이 필요하다.
- 검색 상위는 시총 상위와 거의 부분집합 관계라, 시총 풀 밖에서 합류하는 종목은 0~3개 수준.
- **시세 리스트 파싱(`_fetch_sise_movers`)**: `sise_upper`/`sise_quant`/`sise_rise`/`sise_fall`은
  페이지마다 컬럼이 달라 행 마크업이 제각각이다(상한가는 `class="tltle"`가 없음). 따라서
  순위 데이터 행에만 있는 `<td class="no">` 셀을 앵커로 잡고 직후 첫 종목 링크를 뽑는다.
  네이버가 이 마크업을 바꾸면 정규식 `_SISE_MOVER_ROW`를 조정해야 한다(비공식 엔드포인트).
- **거래량(`sise_quant`)은 KODEX/TIGER 인버스 ETF가 상위를 점령**한다 — `EXCLUDE_NAME_PAT`로
  대부분 걸러지므로 실효 합류 수는 limit보다 훨씬 적을 수 있다(정상).
- `rise`/`fall`(급등·급락)은 전체 시장을 나열해 수백~천 행이라 반드시 `limit`으로 자른다.
  현재는 `upper`/`quant`만 사용하지만, 같은 함수에 `kind`만 바꾸면 즉시 추가 가능.
